# 大数据处理技术课程大作业报告

## 一、大数据环境部署及使用

本次大作业围绕 Linux 与 Hadoop 生态环境的部署、配置与使用展开。报告在保留课程要求的基础上，按照“环境部署总结 + 命令操作说明 + 结果分析”的结构组织内容，重点说明关键配置项的作用、常见命令的使用方法以及各组件在大数据处理流程中的定位。

### （一）大数据环境部署

### 1. 虚拟机与Ubuntu系统安装

实验环境采用 VMware Workstation 创建 Ubuntu 22.04 虚拟机，虚拟机内存配置为 8GB，系统盘分配 80GB。安装过程中首先完成 ISO 镜像挂载、磁盘分区和网络模式设置，然后对系统软件源进行更新，保证后续 Hadoop 生态组件的安装依赖能够正常下载。安装完成后，通过 `hostnamectl`、`ip addr` 和 `java -version` 检查主机名、网络连通性以及 Java 运行时环境是否正常。

{{img_01}}

从部署经验来看，Ubuntu 环境搭建阶段最重要的工作不是单纯完成系统安装，而是提前统一时区、主机名、静态 IP、免密登录和 JDK 环境变量。这样做的原因是 Hadoop 及其周边组件对 SSH、主机解析和 Java 版本较为敏感，若前置环境不一致，后续伪分布模式配置很容易反复报错。

### 2. Hadoop伪分布模式配置

Hadoop 安装完成后，核心工作是修改 `core-site.xml`、`hdfs-site.xml`、`mapred-site.xml` 和 `yarn-site.xml` 四类配置文件。`core-site.xml` 主要定义默认文件系统地址和 Hadoop 临时目录；`hdfs-site.xml` 负责 NameNode、DataNode 的数据块副本数和元数据目录；`mapred-site.xml` 负责指定 MapReduce 的执行框架；`yarn-site.xml` 决定 ResourceManager 与 NodeManager 的通信方式。完成配置后，需要执行 `hdfs namenode -format` 初始化文件系统元数据，再依次启动 HDFS 与 YARN 服务并通过 Web UI 或 JPS 结果检查服务状态。

{{img_02}}

在伪分布模式下，虽然所有角色都运行在同一台虚拟机中，但配置逻辑与分布式环境一致，因此该阶段的重点是理解每个配置文件“管什么、为什么这样写、写错会出现什么现象”。例如，若 `fs.defaultFS` 与主机解析不一致，上传文件到 HDFS 时就会出现连接失败或找不到 NameNode 的问题。

### 3. HBase安装与连接测试

HBase 作为面向列的分布式数据库，适合处理海量、半结构化的数据。安装时需要在 `hbase-site.xml` 中配置 HBase 数据目录、ZooKeeper 地址以及是否与 Hadoop 集群共享存储目录。完成配置后，启动 `start-hbase.sh`，再使用 `hbase shell` 进入交互终端，通过 `status`、`list` 等命令检查服务是否可用。

{{img_03}}

HBase 的配置重点是与 HDFS 的协同关系。因为 HBase 实际上把数据底层存储到 HDFS 中，所以如果 HDFS 服务没有正常启动，或者 HBase 指向了错误的数据目录，即使 HBase Shell 能够打开，也很可能无法完成建表和写入操作。

### 4. Hive安装与元数据配置

Hive 用于在 Hadoop 平台上以类 SQL 的方式完成离线数据处理。部署时首先需要准备 MySQL 元数据库，然后在 `hive-site.xml` 中配置 JDBC 连接串、驱动类、用户名和密码，并初始化 Metastore 模式。完成后，通过 `schematool -dbType mysql -initSchema` 初始化元数据表结构，再使用 `hive` 或 `beeline` 启动客户端进行验证。

{{img_04}}

Hive 的关键配置不仅仅是能否启动客户端，更重要的是元数据连接是否稳定。如果 MySQL 驱动缺失、端口未开放或数据库授权不完整，往往会在建表或查询阶段报出元数据相关异常，因此环境部署阶段应先完成数据库连通性验证。

### 5. Spark运行环境验证

Spark 组件安装完成后，需要在 `spark-env.sh` 中配置 `JAVA_HOME`、`HADOOP_CONF_DIR` 和 `SPARK_MASTER_HOST` 等关键参数，并通过 `spark-shell`、`pyspark` 或 `spark-submit --version` 验证运行环境。若 Spark 需要与 Hive 集成，还要补充 Hive 元数据依赖，使 Spark SQL 可以直接访问 Hive 表。

{{img_05}}

Spark 的优势在于内存计算与统一分析接口。部署阶段的重点是保证 Spark 能够同时读取本地文件和 HDFS 文件，并具备运行 RDD 与 DataFrame 程序的基础条件，这也是后续第二大题“数据分析与可视化”能够顺利完成的前提。

### （二）Linux及Hadoop生态系统组件的使用

### 1. Linux目录与文件操作

在 Linux 命令操作部分，本次实验按课程要求构造了三层目录树，并使用 `mkdir`、`cd`、`pwd`、`ls`、`mv`、`rm -r` 等命令完成目录创建、切换、改名和删除操作。文件部分则使用 `touch`、`vim`、`cp`、`mv`、`cat`、`chmod`、`chown` 等命令完成文件创建、查看、复制、移动、删除与权限调整。该部分的核心在于掌握 Linux 文件系统的组织逻辑和权限模型，而不只是机械记忆命令。

{{img_06}}

通过目录树实验可以发现，规范命名会显著降低后续管理难度。例如，在多级目录中采用“姓名拼音 + 层级编号”的方式，可以快速看出文件所在层级。权限操作则体现出 Linux 多用户环境的管理思想，只有理解读、写、执行三个权限位的意义，才能在真实项目部署中正确设置运行目录。

### 2. HDFS shell命令操作

HDFS 操作部分使用 `hdfs dfs -ls`、`-mkdir`、`-mv`、`-rm -r`、`-put`、`-cat`、`-get` 和 `-cp` 等命令，完成用户目录查看、目录树创建、上传下载以及同名与异名复制等任务。与本地文件系统相比，HDFS 更强调面向分布式环境的数据存储与传输，因此在执行命令时需要关注路径前缀、用户目录以及块存储的逻辑位置。

{{img_07}}

通过上传与下载实验可以验证两个关键点：第一，`-put` 命令是否真正将本地文件写入 HDFS；第二，下载回本地后的结果是否与源文件一致。只有同时检查 HDFS 端与本地端的状态，才能说明数据传输任务真正完成。

### 3. HBase表设计与数据维护

HBase Shell 部分先查看现有表结构，再设计带有两个以上列族的业务表，并插入不少于两行数据。随后完成行数统计、列族添加删除、单元格数据删除以及表删除等任务。由于 HBase 的数据模型不同于传统关系数据库，其重点在于行键、列族和列限定符的设计是否合理，因此本实验在表设计阶段特别强调了字段分组与命名规范。

{{img_08}}

从实验结果看，HBase 适合存放访问模式明确、结构相对灵活的数据。例如，可以把用户基本信息放在 `info` 列族中，把学习行为放在 `study` 列族中。这样既方便按列族管理，又便于后续扩展新的字段。

### 4. HiveQL建库建表与查询

Hive 实验首先创建业务数据库和数据表，然后分别演示从本地文件与 HDFS 文件装载数据，再基于已有表结构创建空表，最后通过插入、分组统计和条件查询完成数据分析。HiveQL 的优点在于可以用接近 SQL 的语法完成离线统计任务，因此在大数据场景中更适合作为仓库查询层。

{{img_09}}

该部分最值得注意的是“表结构设计”和“数据装载路径”之间的匹配关系。如果字段分隔符、存储格式或路径写错，即使数据文件真实存在，Hive 查询结果也可能全部为空值。因此建表前先明确字段类型和分隔符，往往比后续排错更重要。

### 5. Spark RDD与Spark SQL综合分析

Spark 部分分别从内存与文件系统读取数据，利用 RDD 完成基础统计，再通过 DataFrame 与 Spark SQL 完成筛选、分组、排序等分析操作。RDD 更适合展示 Spark 的底层分布式计算逻辑，Spark SQL 则适合完成结构化数据分析，两者结合能够较全面地体现 Spark 在大数据处理中的作用。

{{img_10}}

本部分的重点并不是单个 API 的记忆，而是理解“数据读取 - 转换 - 聚合 - 输出”的分析链路。无论使用 RDD 还是 DataFrame，本质上都是围绕这一链路组织计算任务，因此程序编写时要尽量保持步骤清晰、字段命名统一、输出结果可复核。

## 二、大数据处理

本部分以“在线课程学习行为数据”为示例，按照数据采集、数据清洗、数据存储、数据分析、数据可视化五个环节组织实验流程。考虑到课程要求每个环节都需要给出代码与说明，因此本报告不仅描述处理过程，还在相应位置嵌入了核心程序片段。

### （一）数据采集与数据准备

### 1. 数据采集思路

为了满足“不少于 5000 行”的要求，本次实验使用 Python 生成了 6500 条模拟学习行为原始数据，字段包括用户编号、地区、课程类别、学习时长、订单金额、完成状态、评分和访问日期等。原始数据中故意保留少量重复记录、缺失值和异常值，用于后续展示数据清洗步骤。对应程序文件为 `project/code/generate_raw_data.py`，运行后生成 `project/data/raw_learning_data.csv`。

```python
rows = []
for index in range(6500):
    rows.append({
        "user_id": f"U{100000 + index}",
        "province": random.choice(PROVINCES),
        "course_type": random.choice(COURSE_TYPES),
        "watch_seconds": random.randint(300, 14400),
        "payment_amount": round(random.uniform(19, 399), 2),
        "completion_rate": round(random.uniform(0.2, 1.0), 2)
    })
```

该阶段的目标是准备一份结构完整、字段含义明确、可直接进入清洗流程的原始数据集。即使数据来自模拟生成，也仍然遵循真实业务数据的字段组织方式，这样在后续分析阶段才能得到有意义的统计结果。

### 2. 数据清洗与格式转换

数据清洗程序负责去重、补全缺失值、统一日期格式，并将明显异常的学习时长压缩到合理范围。清洗后的结果保存在 `project/data/clean_learning_data.csv`，同时输出一份 `project/output/cleaning_summary.json` 用于记录清洗前后行数变化。核心脚本为 `project/code/clean_data.py`。

```python
df = df.drop_duplicates()
df["province"] = df["province"].fillna("未知地区")
df["rating"] = df["rating"].fillna(df["rating"].median())
df["watch_seconds"] = df["watch_seconds"].clip(lower=60, upper=14400)
df["visit_date"] = pd.to_datetime(df["visit_date"]).dt.strftime("%Y-%m-%d")
```

从清洗逻辑可以看出，大数据处理并不是“把数据读进程序就算完成”，而是要先解决重复、缺失和格式不统一的问题。只有输入数据质量稳定，后面的 HDFS 存储、PySpark 分析和可视化结果才具有可解释性。

### （二）数据存储与分析实现

### 1. HDFS存储方案

清洗完成后，需要把规范化后的 CSV 文件上传到 HDFS 对应目录，以便后续 Spark 作业直接从分布式文件系统读取。考虑到本次交付同时要求源代码与说明，实验包内单独提供了 `project/code/hdfs_upload_commands.sh`，其中给出了创建目录、上传文件和检查上传结果的完整命令序列。

```bash
hdfs dfs -mkdir -p /course_project/learning_data
hdfs dfs -put -f clean_learning_data.csv /course_project/learning_data/
hdfs dfs -ls /course_project/learning_data
hdfs dfs -cat /course_project/learning_data/clean_learning_data.csv | head
```

这一步骤的意义在于把本地处理后的结构化数据迁移到分布式存储层，使 Spark 能够在统一的数据入口上完成后续计算。通过 `-ls` 与 `-cat` 的组合检查，可以同时验证文件是否上传成功以及内容是否可读。

### 2. PySpark分析实现

分析脚本 `project/code/pyspark_analysis.py` 使用 SparkSession 读取清洗后的 CSV 文件，并从课程类别、地区收入、完课率和日趋势四个维度进行统计。为了方便在无 Spark 环境下仍能得到可检查结果，脚本内部设计了 pandas 回退逻辑：若当前环境未安装 PySpark，则自动切换为 pandas 计算并输出同名结果文件。

```python
category_df = source.groupBy("course_type").agg(
    count("*").alias("order_count"),
    avg("payment_amount").alias("avg_payment")
)

province_df = source.groupBy("province").agg(
    sum("payment_amount").alias("total_payment")
).orderBy(col("total_payment").desc())
```

该程序的设计体现出“大数据工具优先、兼顾交付可复核性”的思路。课程要求强调 PySpark，因此主实现采用 Spark 写法；同时考虑到打包验收时可能没有完整集群环境，又增加了本地回退机制，保证结果文件与图表仍然能够生成。
